### Question 4 ⚠️ Issue — Missing the Most Important Advantage

**"What are the advantages of using Ansible modules over executing commands directly?"**

**Intended answer: "All of the above"** — The question is valid in concept, but it has a significant gap: **idempotency is completely missing**, and that is by far the most critical advantage of using modules over raw commands. Idempotency means running the same module 10 times produces the same result as running it once — the `service` module checks if a service is already running before starting it, the `file` module checks if a directory exists before creating it, and so on. Raw `command` and `shell` tasks are NOT idempotent.

Additionally, "All of the above" as a correct answer is a weak question design — it rewards test-taking strategy over knowledge. A student can often identify "All of the above" as the answer without fully evaluating each option.

**Suggested improvement:** Replace one of the weaker options with idempotency and remove "All of the above":

> - Ansible modules are idempotent — running them multiple times produces the same result
> - Ansible modules handle error checking and complex platform-specific logic
> - Ansible modules abstract away OS-level differences between distributions
> - Ansible modules require you to write custom error-handling code ← (the distractor)

### Question 5 ✅ Good Question, Minor Clarity Issue

**"In what situations would the 'command' module be a better fit than the 'script' module?"**

**Correct answer: "When you need to execute a command on the remote machine directly"** — Conceptually correct, but the phrasing could be sharper. The defining difference is _where the script lives_: `script` takes a script file from the **control machine** and transfers it to the remote node before executing. `command` runs a single command that already exists on the **remote machine**. Option D ("When you need to run a script stored on the control machine") actually describes exactly when you'd use `script`, not `command` — which makes it an effective distractor.

**Minor suggestion:** Rewrite the correct answer to emphasize the location distinction more explicitly: _"When you need to execute a command that already exists on the remote machine"_ — this more cleanly differentiates it from the `script` module's use case.

### Question 6 ⚠️ Issue — Correct Answer Is Too Vague

**"What is the difference between the 'command' and 'script' modules?"**

**Correct answer: "The 'command' module executes commands while the 'script' module runs scripts"** — This answer is technically true but dangerously superficial, and it overlaps significantly with Question 5. A student could get this right without understanding the actual architectural difference between the two modules.

The real distinction is: the `script` module **copies a script from the control node to the remote node** and then executes it there. The `command` module **runs a command that already exists on the remote node** using its path. That location-of-execution distinction is what makes them different tools for different situations.

**Suggested improved answer options:**

> - The `command` module runs a binary/command that exists on the remote host; the `script` module transfers a local script to the remote host and runs it there
> - The `command` module runs on the control node; the `script` module runs on the remote node
> - The `command` module can only run Python scripts; the `script` module handles all script types
> - There is no meaningful difference; both execute arbitrary code remotely

### Question 9 ⚠️ Issue — Redundant with Question 8

**"Which Ansible module would you use to add a line to a file?"**

**Correct answer: `lineinFile`** — As noted above, this is the same knowledge point as Q8. A 10-question quiz should not spend 20% of its questions on a single module, especially when core Ansible concepts like **handlers**, **variables**, **inventory**, and **idempotency** aren't tested at all.

**Suggested replacement question:**

> _"What does a handler in Ansible do?"_
> 
> - It executes at the beginning of every play regardless of task results
> - It executes only when notified by a task, and only once per play even if notified multiple times
> - It is used to define variables for a playbook
> - It manages the order in which hosts are contacted
