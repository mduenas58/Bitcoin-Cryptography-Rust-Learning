**Hashed Time Lock Contract (HTLC)** is a trustless, conditional payment mechanism that combines a **hashlock** and a **timelock**. It ensures that the receiver can claim the funds only by presenting a cryptographic preimage (proof of secret) before a deadline, otherwise the sender can reclaim the funds.

---

## 1. Core Components

| Component      | Description |
|----------------|-------------|
| **Hashlock**   | Locks the funds with the hash of a secret `H(s)`. The secret `s` (preimage) must be revealed to spend the output. |
| **Timelock**   | If the secret is not revealed within a specified time, the funds become refundable to the sender. |

---

## 2. How an HTLC Works (High-Level Flow)

1. **Setup**  
   - The intended receiver (Bob) generates a random secret `s`, computes `h = hash(s)`, and gives `h` to the sender (Alice).  
   - Alice creates a contract that locks funds with `h` and a deadline `T`.

2. **Claim path (successful)**  
   - Bob must reveal `s` before `T`.  
   - The contract verifies `hash(s) == h`. If valid, funds are transferred to Bob.

3. **Refund path (timeout)**  
   - If the deadline `T` elapses and `s` was not revealed, Alice can reclaim the funds.

Because the secret `s` is required to unlock the funds, it acts as a cryptographic proof that Bob has complied with any off-chain agreement (e.g., received a payment on another chain in an atomic swap).

---

## 3. Common Use Cases

- **Atomic Swaps**  
  Cross-chain exchange of assets without a trusted third party. Both sides use HTLCs with the same secret `s` (generated by the initiator); the secret’s revelation unlocks funds on both chains atomically.

- **Payment Channels & Lightning Network**  
  Hashed Time Lock Contracts are the building blocks of multi-hop payments. They allow routing payments across intermediate nodes, each forwarding the transfer once they know the preimage.

## 4. Implementation Example (Solidity on Ethereum)

Below is a simple, self‑contained HTLC smart contract. **This is for educational purposes** – a production‑ready implementation would require careful auditing, handling of re‑entrancy, gas considerations, etc.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract HTLC {
    address public sender;
    address public receiver;
    bytes32 public hashLock;
    uint256 public deadline;
    bool public claimed;

    event LogHTLCDeployed(
        address indexed sender,
        address indexed receiver,
        bytes32 hashLock,
        uint256 deadline
    );
    event LogClaimed(address indexed receiver, bytes32 secret);
    event LogRefunded(address indexed sender);

    // Deployer is the sender, specifies receiver, hash and duration
    constructor(address _receiver, bytes32 _hashLock, uint256 _duration) payable {
        require(msg.value > 0, "Send some ETH");
        require(_duration > 0, "Duration must be > 0");
        sender = msg.sender;
        receiver = _receiver;
        hashLock = _hashLock;
        deadline = block.timestamp + _duration;
        emit LogHTLCDeployed(sender, receiver, hashLock, deadline);
    }

    // Bob reveals the secret to claim the funds
    function claim(bytes32 _secret) external {
        require(msg.sender == receiver, "Only receiver can claim");
        require(!claimed, "Already claimed");
        require(block.timestamp <= deadline, "Deadline expired");
        require(keccak256(abi.encodePacked(_secret)) == hashLock, "Invalid secret");

        claimed = true;
        (bool success, ) = receiver.call{value: address(this).balance}("");
        require(success, "Transfer failed");
        emit LogClaimed(receiver, _secret);
    }

    // Alice can refund after the deadline if not claimed
    function refund() external {
        require(msg.sender == sender, "Only sender can refund");
        require(!claimed, "Already claimed");
        require(block.timestamp > deadline, "Deadline not reached");

        claimed = true;
        (bool success, ) = sender.call{value: address(this).balance}("");
        require(success, "Transfer failed");
        emit LogRefunded(sender);
    }

    // Allow contract to receive ETH (if needed)
    receive() external payable {}
}
```

### How to Use

1. **Bob generates a secret**  
   `s = 0xabc...def` (32 bytes), then computes `h = keccak256(abi.encodePacked(s))`.

2. **Bob gives `h` to Alice**  
   She deploys the contract with `receiver = Bob's address`, `_hashLock = h`, and a `_duration` in seconds, attaching the ETH she wants to lock.

3. **Bob claims**  
   Bob calls `claim(s)` before the deadline. The contract verifies the hash and transfers the balance.

4. **Refund (if Bob doesn’t claim)**  
   After the deadline, Alice calls `refund()` to get the ETH back.

## 5. Implementation in Bitcoin (Script)

Bitcoin implements HTLCs natively in its scripting language using `OP_SHA256`, `OP_EQUALVERIFY`, `OP_CHECKLOCKTIMEVERIFY` (CLTV), etc. A typical HTLC script looks like:

```
OP_IF
    <receiver_pubkey> OP_CHECKSIG
OP_ELSE
    <locktime> OP_CHECKLOCKTIMEVERIFY OP_DROP
    <sender_pubkey> OP_CHECKSIG
OP_ENDIF
```

And the witness/input that satisfies the `OP_IF` branch must also include the preimage that hashes to the required value (often placed in the redeem script using `OP_HASH160` or similar). The Lightning Network uses a more complex construction with commitment transactions and revocation keys, but the underlying conditional logic remains the same.

## 6. Security Considerations

- **Preimage Revelation**  
  Once a party reveals the secret on-chain, it becomes publicly visible. Any observer can also use it to claim other HTLCs secured with the same hash. This is why in atomic swaps both chains must use the **same** secret.

- **Time Drift & Deadline Selection**  
  The deadline must be long enough for the path of the transaction (including cross-chain confirmations) to complete, yet not so long that funds remain locked unnecessarily. A common pattern is to set the **refund deadline** later than the **claim deadline** on the counterparty’s side to give time for dispute resolution.

- **Re‑entrancy & Re‑play**  
  In Ethereum, always follow the “Checks‑Effects‑Interactions” pattern (state updates before external calls) and consider using a re‑entrancy guard if the contract becomes more complex.

- **Secret Management**  
  The secret generator must keep `s` secret until it is safe to reveal it. If it is leaked early, a malicious party can steal funds.

---

## 7. Summary

HTLCs enable trustless conditional payments by coupling a cryptographic puzzle (hashlock) with a timeout (timelock). They are the fundamental primitive behind atomic swaps, cross‑chain interoperability, and scalable payment networks like Lightning. The implementation can be done with a few lines of Bitcoin Script or a small Solidity contract, but always requires careful design of deadlines and handling of secret revelation to avoid vulnerabilities.
