Here is a comprehensive explanation of **SHA-256** (Secure Hash Algorithm 256-bit), covering its purpose, internal workings, security properties, and real-world applications.

---

## 1. What is SHA-256?

SHA-256 is a **cryptographic hash function** – a mathematical algorithm that takes an input (any digital data: a file, a password, a message) and produces a fixed-size, seemingly random output called a **digest** or **hash**.

- **Output size:** 256 bits (32 bytes), usually represented as 64 hexadecimal characters.
- **Part of:** SHA-2 family, designed by the NSA and published by NIST in 2001.
- **Not encryption:** Hashing is one-way; you cannot "decrypt" a hash back to the original input.

**Example** (using an empty string):
```
SHA256("") = 
e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
```

Even a tiny change in input flips the output entirely (avalanche effect).

---

## 2. Core Properties of a Cryptographic Hash

SHA-256 must satisfy:

| Property | Meaning |
|----------|---------|
| **Deterministic** | Same input → same hash (always). |
| **Fast computation** | Hash is quick to calculate for any input size. |
| **Preimage resistance** | Given hash `h`, impossible to find *any* input that hashes to `h` (except brute-force guessing). |
| **Second preimage resistance** | Given input `m1`, cannot find different `m2` with `SHA256(m1) = SHA256(m2)`. |
| **Collision resistance** | Cannot find *any two* different inputs `m1, m2` with same hash (birthday attack resistance: ~2^128 effort). |
| **Avalanche effect** | Flipping one input bit changes ~50% of output bits unpredictably. |

## 3. How SHA-256 Works Internally (Step-by-Step)

SHA-256 processes data in **512-bit chunks**. It’s a **Merkle-Damgård** construction with a compression function.

### Step 1: Padding
Append bits to make total length ≡ 448 mod 512, then add the original length as a 64-bit integer.  
- Always add a `1` bit, then zeros, then 64-bit length.  
- Result: message length is a multiple of 512 bits.

### Step 2: Initialize Hash Values (H0–H7)
Eight 32-bit constants derived from the fractional parts of the square roots of the first 8 primes:
```
h0 = 0x6a09e667
h1 = 0xbb67ae85
h2 = 0x3c6ef372
h3 = 0xa54ff53a
h4 = 0x510e527f
h5 = 0x9b05688c
h6 = 0x1f83d9ab
h7 = 0x5be0cd19
```

### Step 3: Prepare Message Schedule (for each 512-bit chunk)
Break chunk into 16 words (each 32 bits): `w[0] ... w[15]`  
Extend to 64 words (`w[0]` to `w[63]`) using:
```
for i from 16 to 63:
    s0 = (w[i-15] rightrotate 7) XOR (w[i-15] rightrotate 18) XOR (w[i-15] rightshift 3)
    s1 = (w[i-2] rightrotate 17) XOR (w[i-2] rightrotate 19) XOR (w[i-2] rightshift 10)
    w[i] = w[i-16] + s0 + w[i-7] + s1
```
(All operations are 32-bit unsigned integer arithmetic, modulo 2^32.)

### Step 4: Initialize Working Variables
```
a = h0, b = h1, c = h2, d = h3, e = h4, f = h5, g = h6, h = h7
```

### Step 5: Main Compression Loop (64 rounds per chunk)
For each round `i` from 0 to 63:
```
Σ1 = (e rightrotate 6) XOR (e rightrotate 11) XOR (e rightrotate 25)
Ch = (e AND f) XOR ((NOT e) AND g)
temp1 = h + Σ1 + Ch + k[i] + w[i]
Σ0 = (a rightrotate 2) XOR (a rightrotate 13) XOR (a rightrotate 22)
Maj = (a AND b) XOR (a AND c) XOR (b AND c)
temp2 = Σ0 + Maj

h = g
g = f
f = e
e = d + temp1
d = c
c = b
b = a
a = temp1 + temp2
```
**Constants k[i]** are 32-bit fractions of cube roots of first 64 primes.

### Step 6: Add to Hash Value
After all rounds:
```
h0 = h0 + a
h1 = h1 + b
... (all 8 variables)
```

### Step 7: Final Output
After processing all chunks, concatenate the final `h0` through `h7` (each as 32-bit hex) to produce a 256-bit digest.

## 5. Security & Weaknesses

- **No known practical collision attack** (as of 2025). Best attack reduces 2^256 to ~2^64.5 for 31 rounds, but full 64 rounds remains secure.
- **Length extension attack** possible because SHA-256 is a Merkle-Damgård hash.  
  *Mitigation:* Use HMAC-SHA256, not raw SHA256(key || message).
- **Not suitable for password storage** alone (too fast → brute-force). Use with salt + key stretching (PBKDF2, bcrypt, Argon2).

## 6. Real-World Applications

| Use Case | How SHA-256 is used |
|----------|----------------------|
| **Bitcoin & Blockchain** | Proof-of-work (double SHA256 of block header). Also addresses and Merkle trees. |
| **TLS/SSL certificates** | Certificate signature verification (together with RSA/ECDSA). |
| **File integrity checks** | Compare hash of downloaded file against known good hash. |
| **Git** | Commit IDs (SHA-1 historically; SHA-256 now in transition). |
| **Password hashing** | Not directly – but used in HMAC for key derivation. |
| **Digital signatures** | Hash the message, then sign the hash (much faster). |
| **Deduplication** | Hash chunks of data to find duplicates. |

---

## 7. Performance & Implementation

- **Software speed:** ~200–400 MB/s per core (modern CPU).  
- **Hardware acceleration:** Intel SHA extensions (SHA-NI) improve speed ~3-5x.  
- **Memory usage:** Very low – only a few hundred bytes of state.  
- **Typical implementations:** OpenSSL, Crypto++, Java `MessageDigest`, Python `hashlib`.

Example (Python):
```python
import hashlib
hash = hashlib.sha256(b"Hello, world!").hexdigest()
print(hash)  # 315f5bdb76d078c43b8ac0064e4a0164612b1fce77c869345bfc94c75894edd3
```

## 8. Variants & Comparison

| Algorithm | Output bits | Security (collision) | Speed (relative) |
|-----------|-------------|----------------------|------------------|
| SHA-1 | 160 | Broken (2^63) | Fast |
| SHA-256 | 256 | Secure (>2^128) | Medium |
| SHA-512 | 512 | Secure (>2^256) | Fast on 64-bit |
| SHA-3 (Keccak) | variable | Secure | Slower than SHA-2 |
| BLAKE2/3 | variable | Secure | Very fast |

**Note:** On 64-bit CPUs, SHA-512 is often *faster* than SHA-256 because it uses 64-bit words and has fewer rounds per byte.

---

## 9. Common Misconceptions

- ❌ "SHA-256 is encryption" – No, it’s a one-way hash.  
- ❌ "You can reverse SHA-256" – Only by brute-force guessing (infeasible for any real secret).  
- ❌ "Two different files can’t have the same hash" – They can (pigeonhole principle), but finding such a pair is practically impossible.  
- ❌ "Adding salt makes SHA-256 secure for passwords" – Not enough; use a slow KDF.

---

SHA-256 is a workhorse of modern cryptography – well-studied, hardware-accelerated, and widely trusted. While newer hashes like SHA-3 or BLAKE3 exist, SHA-256 remains the gold standard for most applications requiring collision resistance and preimage resistance. Always use it correctly (e.g., HMAC for keys, not for password storage) and stay updated on cryptanalysis progress.
